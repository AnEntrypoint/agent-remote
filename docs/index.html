Remote shell.
No servers. No ports.

Run commands on any machine, anywhere, using a shared seed phrase. Traffic is end-to-end encrypted. NAT is traversed automatically. Nothing to configure.

bun x agent-remote --server --key myseed
bun x agent-remote --run --key myseed whoami

Quick start

  • On the remote machine — start the shell server

    bun x agent-remote --server --key myseed

    Prints the public key. Stays listening on the DHT.

  • On your local machine — start the daemon

    bun x agent-remote --daemon

    Starts a local HTTP server. Pays the DHT bootstrap cost once and keeps the node alive.

  • Run commands atomically

    bun x agent-remote --run --key myseed whoami
bun x agent-remote --run --key myseed "df -h"
bun x agent-remote --run --key myseed ls /var/log

    Each call is a single RPC round-trip. The daemon stays warm between calls.

  • Shutdown when done

    bun x agent-remote --shutdown

How it works

Each --run call walks this pipeline:

CLI --run ↓ POST /rpc { method: "run", params: { key, cmd } } local daemon ↓ sha256(seed) → DHT.keyPair(32 bytes) key derivation ↓ DHT node.connect(publicKey) → UDP holepunch HyperDHT socket ↓ Noise XX handshake → encrypted channel Protomux { protocol: "hypershell" } ↓ ShellClient sends: [shell, flag, cmd] remote ShellServer ↓ spawns $SHELL → pipes stdout / stderr stream back ↓ channel close → { stdout, stderr } result returned to CLI

Key derivation

The seed phrase is SHA-256 hashed into 32 bytes, then fed into DHT.keyPair() to produce a Curve25519 keypair. The public key is the server's address on the DHT — no DNS, no IPs, no port forwarding. 

sha256("myseed") → 32 bytes → { publicKey, secretKey }
                                       ↑
                             server listens here
                             client connects here

Shell detection

The daemon reads the shell from the environment at startup — once. Whatever shell launched the daemon is used for every command.

PlatformSourceDefault
Windows%COMSPEC%cmd.exe /c
Unix$SHELL/bin/sh -c

Daemon architecture

The daemon is a long-lived HTTP server on 127.0.0.1 at a random port, written to $TMPDIR/agent-remote.port. Each CLI invocation reads that file to find the daemon. This means DHT bootstrap happens once — subsequent --run calls are just local HTTP round-trips followed by a DHT connection.

RPC API

The daemon exposes a JSON-RPC interface. Use it directly from any language.

MethodParamsReturns
run{ key, cmd }{ stdout, stderr }
connect{ key }{ id }
disconnect{ id }{ ok }
connections—{ ids }
shutdown—{ ok }
import fs from 'fs'
import os from 'os'
import path from 'path'

const port = fs.readFileSync(path.join(os.tmpdir(), 'agent-remote.port'), 'utf8').trim()

const res = await fetch(`http://127.0.0.1:${port}/rpc`, {
  method: 'POST',
  body: JSON.stringify({ id: 1, method: 'run', params: { key: 'myseed', cmd: 'whoami' } })
})

const { result } = await res.json()
console.log(result.stdout) // → "myuser\n"

Security

What is protected

All traffic is end-to-end encrypted via the Noise Protocol (XX handshake). The DHT relay nodes cannot read your data — they only route encrypted packets. Both sides must present a valid keypair derived from the Curve25519 exchange to complete the handshake.

What to pay attention to

Open firewall by default. The server uses firewall: () => false, which accepts connections from any DHT keypair. Anyone who knows (or brute-forces) a valid connection can open a shell. The seed is your only access control — treat it like a password.
The seed is a root credential. Anyone with the seed can derive the keypair and connect. Use a long random seed: openssl rand -hex 16. Never commit it. Never reuse it across machines.
Local daemon is trusted. The RPC server binds only to 127.0.0.1 — not reachable from the network — but any local process can call it. On shared machines, treat the port file as a sensitive credential.

Make a hard seed

The seed is your only access control. Generate one you can't guess:

openssl rand -hex 16
# → e3b4f2a1c8d7e6f5a4b3c2d1e0f9a8b7

bun x agent-remote --server --key e3b4f2a1c8d7e6f5a4b3c2d1e0f9a8b7
